# Assignment Explanation — For the Intern

## What is this assignment about?

At Tapp, we are building **SDKs** (Software Development Kits). An SDK is a package of code that other developers install into their own apps to add features — for example, our widgets SDK.

When an SDK is updated, things can break:
- The app that uses the SDK might stop working.
- Functions might change their names or behavior.
- The documentation might no longer match what the code actually does.

Your job in this assignment is to play the role of a Tapp developer before the SDK is sent to the client — someone whose job is to catch these problems before they reach real users.

---

## What will you be doing?

You'll work with three things:

### 1. The SDK (`my-sdk/`)
This is a small JavaScript library with a few simple functions — things like adding numbers, greeting users, validating emails. Think of it as a toolbox that an app can use.

### 2. The Demo App (`my-app/`)
This is a small program that uses the SDK. It calls the SDK's functions and prints the results. Think of it like a test drive of the toolbox.

### 3. The Documentation (`docs/api-reference.md`)
This is a document that describes what the SDK is supposed to do — which functions exist, what inputs they take, and what results they give back. Think of it like the instruction manual for the toolbox.

---

## The problem

**The documentation, the SDK, and the app don't all agree with each other.**

Some things are broken. Some things are subtly wrong. Your mission is to find every mismatch and clearly report what you found.

---

## What skills does this test?

This assignment is designed to see how you approach problems. Specifically:

| Skill | How it shows up |
|---|---|
| **Following instructions** | Can you set things up step by step without skipping ahead? |
| **Reading error messages** | When something breaks, do you read the message or just say "it doesn't work"? |
| **Attention to detail** | Can you catch small differences between what the docs say and what the code does? |
| **Hands-on testing** | Do you actually run the code to verify, or do you only read and guess? |
| **Problem solving** | When you find a bug, can you figure out the fix? |
| **Clear communication** | Can you explain what you found in a way that someone else can understand? |

## How to approach this assignment

1. **Read the whole README first** before you start doing anything. Understand all 5 stages.
2. **Go in order.** Stage 1 before Stage 2, Stage 2 before Stage 3, etc.
3. **When something breaks, don't panic.** Read the error message carefully. Google the error if you need to. Try to understand it before jumping to a fix.
4. **Write things down as you go.** Don't try to remember everything and write reports at the end. Take notes during each stage.
5. **Actually run the code.** The most important thing in this assignment is that you test things by running them, not just by reading the code. Some issues can only be found by running the functions and looking at the actual output.
6. **Be honest.** If you're stuck on something, say so. If you're not sure about something, write your best guess and explain why. We value honesty over perfection.
